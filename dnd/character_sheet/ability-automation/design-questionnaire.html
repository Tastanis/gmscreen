A. Workflow & JSON authoring

A1
Where will automation JSON come from primarily?
Determines whether we invest more in LLM tooling or hand-edit ergonomics.
My default if blank: Mix. The format spec doc is the long-term win for LLM-assisted authoring.
A2
When you paste bad JSON, what should happen?
Strict = JSON gets validated against a schema, errors listed clearly. Lenient = unknown fields silently dropped, runtime tries its best.
My default if blank: Warn but save. Strict frustrates LLM-generated drafts; lenient hides bugs.
A3
Should there be a "format spec" document you can paste into an LLM along with an ability text to get JSON back?
A single Markdown file with the schema, all enums, and 8-10 worked examples covering the common ability patterns.
My default if blank: Yes. Path: dnd/character_sheet/ability-automation/AUTHORING.md. Acts as both LLM context and your reference.

B. Top-level JSON shape

B1
What's the outer shape of an automation JSON?
Today: flat array of cards. The book has clear sections (target, roll, effect, trigger, persistent). A sectioned object would match the book.
My default if blank: Hybrid. Ordered list of typed blocks gives you both: explicit ordering for "effect before vs after roll" cases, and named sections so blocks know their shape.
B2
Should the automation JSON include the ability's name and description?
You said the character sheet stays the source of truth for what the player sees. So this asks whether the JSON ALSO carries the display text (for the inspector / chat output) or only behavior.
My default if blank: Include name only. Avoids drift between sheet text and JSON, but lets the inspector and chat say "Brutal Slam: …" without needing a sheet lookup.
B3
Action type (main/maneuver/move/triggered/free-triggered) and resource cost — where do they live?
The character sheet already has fields for these. Question is whether JSON duplicates them or just consumes them.
My default if blank: Sheet only. Less duplication, less drift. Spend riders inside JSON are different — those stay in JSON since they're per-effect.

C. Targets

C1
How is the ability's distance/range encoded?
Book forms: Melee 1, Ranged 5, Melee 1 or ranged 5, Self, 5 burst, 3 aura, 2 cube within 5, 10×2 line within 5, 4 wall within 10.
My default if blank: Parsed structure. Range enforcement on the board (legal-target highlight, template-placement bounds) needs structured data.
C2
Confirm the target predicate enum.
What can be in the "target =" line. Check what should be supported now; uncheck to drop; add custom in the textarea.
C3
Should target groups be NAMED so later effects can reference specific groups?
Useful when one ability hits everyone in an area but only pushes the primary, or marks just one of several. Example: { "target": "primary", "effects": [...] } vs { "target": "inArea", "effects": [...] }.
My default if blank: Yes. The book has too many "primary takes X, secondaries take Y" patterns to skip this. Adds zero cost when unused (default = "all current targets").
C4
"Optional" target — what does that mean at runtime?
Today a target marked optional silently skips. Book usage: "you can target yourself or one ally" — the ally is optional but the ability still runs.
My default if blank: Always show explicit buttons. Mandatory targets only show "Cancel automation" (not "Skip"). Optional targets show both.

D. Power roll & tiers

D1
"Strongest" attribute — when is it resolved?
Today it resolves at runtime to the source character's highest M/A/R/I/P. If you change attributes between sessions, the ability follows you.
My default if blank: Either. Most book abilities specify a fixed attribute (e.g. "Power Roll + Might"). "Strongest" is for class abilities that explicitly use highest characteristic. Both forms in JSON cover both cases.
D2
Edges and banes — where do they come from at runtime?
Edges/banes alter tier outcomes. They can come from the ability itself, the situation (cover, advantage), and class features.
My default if blank: Both. Most situational adjustments come from the table; the ability sometimes declares its own edge/bane (e.g. "this ability has a double edge"). Pre-roll widget shows current count with +/- buttons.
D3
What effect kinds can appear inside a tier result?
Per the book, tier text is "damage; rider; rider". Riders can be forced movement, conditions via potency, or even cascading abilities ("an ally uses a strike as a free triggered action"). What should the schema allow inside a tier?
My default if blank: Full registry. The book uses cascading abilities, marks, swaps, etc. inside tier text. Restricting now will force schema bumps later.
D4
Critical hits (natural 19/20) — how should the system handle them?
Book rule: nat 19/20 on a 2d10 main-action ability roll grants an immediate extra action. Affects automation because tier 3 ≠ crit.
My default if blank: Badge. Granting the action is a player decision and table-context-heavy; just notify so they don't miss it.

E. Potency riders

E1
How is a potency threshold stored in JSON?
Book reads "M<w" / "P<v" / "I<s". These resolve to character-specific integer thresholds at runtime (a level-1 caster's strong = ~2, a level-10 caster's strong = ~5).
My default if blank: Qualitative with optional override. 99% of abilities are qualitative; the rare numeric override (e.g. "M<1 regardless of caster level") stays expressible.
E2
What can a potency rider do on failure?
Today only conditions / damage. Book has riders that push, slide, force a free strike, force the target to use an ability, etc.
My default if blank: Any effect from the registry. Treat the failure branch as just another effect list.

F. Effects catalog

F1
Confirm the effect kinds we'll implement.
Check the ones to support now. Each becomes a typed "effect" the JSON can use anywhere (tier rider, flat effect block, potency failure, spend bonus).
F2
Damage type enum.
Used to look up immunity/weakness on the receiving token. Uncheck any to drop, add custom in textarea.
F3
Condition enum.
Names the runtime knows how to display and (eventually) auto-track on tokens. Marks are deferred per your earlier note.
F4
Duration enum.
Tag a condition (or any timed effect) with one of these. Runtime can later prompt for saves at end-of-turn.
F5
Forced movement verbs.
All take an integer distance. Vertical variants matter for cliffs/flying.
Bonus: Should distance support "up to N" (player picks 0..N at runtime) in addition to fixed N? I'd say yes — book has both.

G. Triggers / persistent / spend

G1
Are triggered abilities (free triggered actions, reactions) part of THIS pass?
Book has triggered abilities ("Trigger: target takes damage", etc.). Today: zero scaffolding. Adding them well requires runtime hooks on damage/move/turn events.
My default if blank: Author them but runtime is manual. Schema and chat output ready; the auto-detect (event bus on damage/move/turn) is a meaty separate project.
G2
Persistent zones / sustains (auras, walls, vortex, "Persistent N" re-fire) — this pass?
My default if blank: Schema-only for now. Real tracking needs a per-encounter zones registry on the board, which is a bigger lift than the JSON rebuild.
G3
"Spend N Resource: bonus" riders — when do they prompt?
Book has many of these (Spend 1 Wrath, Spend 1 Essence, etc.). Two natural points: before the roll (so they can edge it) or after (apply riders to the result).
My default if blank: Hybrid. Some spends modify the roll (must be upfront); others modify the result (must be after). JSON tags each with a timing.

H. Runtime UX

H1
After the dice roll, how is the tier result chosen?
My default if blank: Auto-pick + confirm. Today's behavior. Edges/banes can shift the effective tier per-target — confirm step lets the user override per-target.
H2
Multi-target token selection — how does the user end it?
When the ability targets "Three creatures" or "up to 3 enemies".
My default if blank: Both. Book has both ("Three allies" exact, "up to 3 enemies in burst" up-to). Flag in JSON, runtime renders accordingly.
H3
Skip behavior on a target prompt.
My default if blank: Depends. Skip is only valid for optional targets. Mandatory targets only offer Cancel.
H4
Should a "Cancel automation" button always be visible during runtime?
My default if blank: Always. Today the runner can hang on the chat side if the user closes the power-roll modal without accepting; explicit cancel everywhere prevents that.
H5
Damage application — auto vs confirm.
My default if blank: Auto-apply. Post a chat message that has Undo. Today's behavior plus Undo would be ideal.
H6
Hidden tokens / chat visibility.
If the target is a hidden NPC, do players see the damage number?
My default if blank: Hide numbers. The point of hidden NPCs is players don't know they exist.
